### 一、 项目简介 (Project Overview)

**项目名称（暂定）：** SmartQuery AI — 智能双模数据库查询与分析系统

**项目定位：**
本系统是一款连接“人”与“数据”的智能桥梁软件。它提供**“专业SQL模式”**与**“自然语言对话模式”**双轨并行功能。专业技术人员可以直接编写、调试和优化SQL；而业务人员无需掌握任何SQL语法，只需像聊天一样提出中文需求（例如：“帮我统计上个月华东地区销售额前十的产品”），系统即可自动理解意图、生成标准SQL、执行查询，并将复杂的数据即时转化为直观的图表（折线图、柱状图等）。

**核心受众：**
*   **非技术人员（产品、运营、分析师、管理层）：** 无需代码背景，分钟级获取业务指标，打破数据孤岛。
*   **技术人员（DBA、后端开发、数据工程师）：** 用于快速预览、辅助编写复杂SQL、生成测试数据，提升研发能效。

### 二、 项目目标 (Project Goals)

#### 1. 功能目标 (Functional Goals)
*   **双模式自由切换：** 
    *   *SQL模式：* 支持主流数据库（MySQL, PostgreSQL, ClickHouse等）连接，具备语法高亮、自动补全、执行计划分析等标准IDE功能。
    *   *对话/自然语言模式：* 基于大语言模型（LLM）的 **Text-to-SQL** 技术，精确率达 90% 以上的标准查询。
*   **自动图表化与分析（Data-to-Viz）：** 自动根据查询结果的维度（时间、分类、数值）匹配最适合的图表：趋势图、漏斗图甚至生成一句话的数据洞察。
*   **语义层/元数据管理：** 允许管理员为表名、列名添加中文备注。例如把 `sys_user.u_status` 解释为“用户活跃状态”，帮助大模型更准确地理解业务指标。
*   **多人协作与权限分配：** 建立基于角色的访问控制（RBAC），确保不同层级员工只能查询各自权限范围内的数据。

#### 2. 非功能开发目标 (Non-Functional Goals)
*   **低延迟、高响应：** 自然语言转SQL及生成图表的端到端延迟控制在 3 秒以内。
*   **高安全性：** 确保底线安全，防止AI生成可能破坏数据的恶意SQL，防止越权查询。
*   **良好的用户体验（UX/UI）：** 类似 ChatGPT 与 DataGrip 结合的现代交互界面，支持对话流与工作流的混合操作。

### 三、 需要注意的地方 (Key Considerations & Challenges)

这是此类项目在实际落地中**最容易踩坑**的几个关键领域，需要在系统架构设计之初就做好考量：

#### 1. 安全底线 (Absolute Security)
*   **只读权限（Read-Only Constraint）：** 
    对“自然语言模式”绑定的数据库账号，**绝对**只能授予 `SELECT` 权限。严禁授予 `UPDATE`、`DELETE`、`DROP` 等修改和删除权限，防止AI生成破坏性指令或人为的社工库隐患。
*   **SQL 注入与恶意注入防范：** 
    用户可能会输入：“请帮我把 user 表全部删掉”。虽然底座模型可能拒绝，但用户也可能采用越狱提示词。后端应在执行前使用 **SQL AST（抽象语法树）分析器** 过滤任何可能包含修改、高危函数、跨库查询的语句。
*   **敏感数据脱敏：** 
    对于身份证号、手机号、薪资等敏感信息，系统应能自动识别并进行掩码脱敏（Masking），不直接暴露给业务用户。

#### 2. “大模型幻觉”与 Text-to-SQL 准确率
*   **Schema 提示词优化 (Prompt Engineering + RAG)：** 
    如果数据库表非常多（比如几百张表），直接把所有表结构丢给大模型会导致：1. Token 耗尽或昂贵；2. 模型理解混乱。
    *   *对策：* 引入数据库 Schema 检索机制（利用 RAG 架构），先根据用户问题匹配最相关的几张表和字段，再把精简后的上下文喂给大模型。
*   **别名与中文词典（语义层设计）：**
    大模型无法凭空猜出 `t_ord_dt = 1` 代表“订单已完成”。你需要建立一个精简的语义图谱或元数据词典，定期同步表注释（Comment），给大模型提供足够的背景知识。
*   **“黑盒”确认机制（Human-in-the-Loop）：**
    因为大模型有幻觉，直接展示结果可能会误导决策。**设计上，务必在执行前把 SQL 以及大模型对该 SQL 的原理解释展示出来**（例如：“我为您生成了如下查询：按地区分组，统计了 5 月份的订单和...”），由用户确认无误后再点击“运行”。

#### 3. 大模型选型与成本控制 (Model Selection & Cost)
*   **推理成本：** 频繁使用 GPT-4/Claude 3.5 级别的大模型进行 Text-to-SQL 翻译，在调用量高时成本巨大。
*   **混合精度方案：** 建议使用混合架构。简单问句使用轻量、廉价的高性价比模型（甚至本地 finetune 的小模型，如 Llama-3-8B 衍生微调版）；对于长表关联（Join）或嵌套子查询，再路由给闭源强逻辑模型。

#### 4. 方言与环境适应性 (Database Dialect)
*   不同数据库语法细节相差甚远。MySQL 的 `LIMIT`，在 SQL Server 是 `TOP`，在 Oracle 是 `ROWNUM`。
*   在向大模型请求时，**必须明确指定目标数据库类型和版本**（如：“你是一个 PostgreSQL 15 专家，请根据以下 Schema 生成标准的 PostgreSQL SQL...”）。

#### 5. 无法对齐的业务逻辑（“业务口径”统一）
*   不同公司的“净利润”、“活跃用户”计算公式差异巨大。
*   *解决思路：* 引入中介**语义层/指标层（Metric Layer，类似于 dbt）**。把复杂的、变动的业务口径固化在视图（View）或语义定义文件中，大模型只需针对已经定义好的“宽表”或“指标”进行查询，而不是每次都去尝试用原始底表计算复杂的业务逻辑。
